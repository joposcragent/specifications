# Асинхронные задачи оркестратора

Здесь описаны **алгоритмы выполнения Celery-задач** внутри сервиса `celery-orchestrator` (воркеры обрабатывают сообщения из брокера Redis). Постановка этих задач в очередь идёт через REST-контроллер `queue-broker` (см. [rest.md](./rest.md) и [openapi.yaml](./openapi.yaml)).

Поведение сквозного процесса задано в [UC-02 «Оркестрация процесса»](../../../solution/src/md/40-solution/10-use-cases/UC-02-collection-job.md); стек Celery+Redis — в [ADR-003](../../../solution/src/md/40-solution/90-ADR/ADR-003-celery-redis.md).

**Примечание.** Периодический запуск пакета сбора «раз в час» (UC-02 п.1) обычно настраивается через **Celery Beat** и приводит к постановке в брокер сообщения, которое обрабатывает задача `tasks.collection_batch`; сам Beat — это планировщик, а не отдельная задача из таблицы ниже.

## Сводная таблица задач

| Имя Celery-задачи                 | Роль в UC-02                                                                      |
|-----------------------------------|-----------------------------------------------------------------------------------|
| `tasks.collection_batch`          | П.1–2: пакетный сброс, загрузка поисковых запросов, порождение `collection-query` |
| `tasks.collection_query`          | П.2–3: вызов crawler, ожидание прогресса и завершения по запросу                  |
| `tasks.collection_query_progress` | П.4.2, запись на диаграмме: прогресс по страницам выдачи                          |
| `tasks.collection_query_complete` | П.4.3: завершение сбора по запросу, разблокирование ожидающей логики              |
| `tasks.evaluation`                | П.5–6: запуск оценки вакансии, ожидание `evaluation-complete`                     |
| `tasks.evaluation_complete`       | П.7: фиксация результата оценки, разблокирование ожидающей логики                 |
| `tasks.notification`              | П.7.1: доставка уведомления при высокой релевантности                             |

Ниже под «хранилищем оркестрационных задач» понимается то же хранилище, что и для REST `tasks` (см. [readme.md](./readme.md)): записи типов `COLLECTION_BATCH`, `COLLECTION_QUERY`, `EVALUATION` и т.д.

## `tasks.collection_batch`

**Назначение.** Обработать запуск очередного пакета сбора: получить список поисковых запросов и для каждого инициировать цепочку сбора по одному запросу.

### Вход (аргументы сообщения / kwargs)

| Параметр        | Описание                                      |
|-----------------|-----------------------------------------------|
| `triggeredAt`   | Опционально; момент срабатывания расписания   |
| `correlationId` | Опционально; идентификатор трассировки пакета |

**Алгоритм работы:**

1. Найти и переиспользовать оркестрационную задачу типа `COLLECTION_BATCH` в хранилище со статусом `RUNNING`, зафиксировать `triggeredAt` / `correlationId` в `initial_context` при наличии.
2. Выполнить запрос к **settings-manager**: `GET /search-query/list` (см. UC-02, узел `APP->>SET`).
3. Если список запросов пуст:
   1. Зафиксировать в логе и в контексте пакета отсутствие работы;
   2. Перевести оркестрационную задачу `COLLECTION_BATCH` в терминальный статус `FAILED`;
   3. Завершить Celery-задачу.
4. Для **каждого** элемента `list[].query` из ответа:
   1. Поставить в брокер сообщение, приводящее к запуску `tasks.collection_query`, передав в kwargs как минимум: текст `searchQuery`, UUID созданной задачи пакета `parentBatchTaskUuid` (идентификатор оркестрационной задачи `COLLECTION_BATCH` из шага 1).
5. После постановки всех дочерних сообщений обновить контекст пакета (число порождённых запросов) и оставить статус пакета `RUNNING` до агрегированного завершения всех дочерних `COLLECTION_QUERY` (политика агрегирования — реализация; см. readme по REST `tasks`).
6. Завершить Celery-задачу `collection_batch` успешно (пакет не обязан дожидаться в этом же вызове завершения всех дочерних задач — они выполняются асинхронно).

**Ошибки**

- Недоступность settings-manager или ошибка HTTP: залогировать, перевести задачу пакета в `FAILED` при необходимости, пробросить или подавить исключение в соответствии с политикой retry Celery.
- Ошибка постановки отдельного `collection_query` в брокер: залогировать, по возможности продолжить остальные запросы или прервать пакет — задаётся реализацией.
- Пустой массив `list[].query` в ответе от settings-manager

## `tasks.collection_query`

**Назначение.** Запустить сбор вакансий по одному поисковому запросу через **crawler-headhunter** и дождаться завершения по событию из брокера (событийная модель UC-02 п.3–4).

| Параметр                  | Описание                                                                                                                  |
|---------------------------|---------------------------------------------------------------------------------------------------------------------------|
| `searchQuery`             | Строка поискового запроса                                                                                                 |
| `parentBatchTaskUuid`     | UUID оркестрационной задачи `COLLECTION_BATCH`                                                                            |
| `collectionQueryTaskUuid` | UUID оркестрационной задачи `COLLECTION_QUERY` (если создана до постановки в очередь; иначе создаётся в начале алгоритма) |

**Алгоритм работы:**

1. Убедиться, что в хранилище существует запись `COLLECTION_BATCH` с `uuid` = `parentBatchTaskUuid`; при отсутствии — завершить с ошибкой и пометить задачу как `FAILED`.
2. При отсутствии заранее выделенного UUID: создать оркестрационную задачу типа `COLLECTION_QUERY` с `parent_task_uuid` = `parentBatchTaskUuid`, статус `RUNNING`, в `initial_context` сохранить `searchQuery`.
3. Вызвать **crawler-headhunter**: `POST /crawler/start` с телом, содержащим поисковый запрос и идентификатор «job» для корреляции — в качестве него использовать `collectionQueryTaskUuid` (как на диаграмме UC-02: `query, job uuid`).
4. Перевести выполнение в режим **ожидания** сигнала завершения по данному запросу:
   1. Промежуточные события `collection-query-progress` обрабатываются отдельной задачей `tasks.collection_query_progress` (обновление БД / прогресса оркестрационной задачи);
   2. Итог приходит через обработку `tasks.collection_query_complete`, которая должна снять блокировку ожидания для данной `collection_query` (механизм: Redis-ключ, chord callback, БД-флаг «complete_received» и т.п. — деталь реализации).
5. После получения сигнала завершения (лог и результат от crawler):
   1. При необходимости обновить оркестрационную задачу `COLLECTION_QUERY` через внутренний вызов логики REST `POST /tasks/{taskUuid}/completion` или эквивалент;
   2. Завершить Celery-задачу успешно.

**Ошибки**

- Ошибка HTTP crawler (4xx/5xx): зафиксировать, завершить оркестрационную `COLLECTION_QUERY` как `FAILED`, пробросить исключение для retry по политике.

## `tasks.collection_query_progress`

**Назначение.** Обработать очередное сообщение о промежуточном прогрессе сбора (между страницами выдачи hh.ru), синхронно с UC-02 п.4.2 и записью прогресса на диаграмме.

| Параметр                    | Описание                                       |
|-----------------------------|------------------------------------------------|
| `collectionQueryTaskUuid`   | UUID оркестрационной задачи `COLLECTION_QUERY` |
| `currentPage`, `totalPages` | Опционально                                    |
| `executionLogFragment`      | Опционально                                    |
| `detail`                    | Опционально; произвольный JSON                 |

**Алгоритм работы:**

1. Проверить наличие оркестрационной задачи с `uuid` = `collectionQueryTaskUuid` и типом `COLLECTION_QUERY`; при отсутствии — залогировать ошибку и выйти.
2. Если задача уже в терминальном статусе — не обновлять прогресс (идемпотентный no-op или лог предупреждения), залогировать ошибку и выйти.
3. Обновить хранилище: записать фрагмент прогресса (`POST /tasks/.../progress` или прямой доступ к БД), обновить `updated_at`, при необходимости агрегировать лог.
4. Успешно завершить Celery-задачу.

**Ошибки**

- Сбой хранилища: retry по настройкам Celery; после исчерпания попыток — сообщение в DLQ или лог критической ошибки (реализация).

## `tasks.collection_query_complete`

**Назначение.** Зафиксировать завершение сбора по запросу и разблокировать ожидающую `tasks.collection_query` (UC-02 п.4.3).

| Параметр                  | Описание                                       |
|---------------------------|------------------------------------------------|
| `collectionQueryTaskUuid` | UUID оркестрационной задачи `COLLECTION_QUERY` |
| `executionLog`            | Лог выполнения                                 |
| `result`                  | Структурированный итог сбора                   |

**Алгоритм работы:**

1. Проверить существование оркестрационной задачи `COLLECTION_QUERY` с данным UUID.
2. Сохранить в хранилище итоговый лог и результат (эквивалент `POST /tasks/{taskUuid}/completion` с `terminalStatus` = `SUCCEEDED`, если crawler сигнализирует об успехе; при ошибке из `result` — `FAILED`).
3. Установить сигнал ожиданию, на котором заблокирована соответствующая `tasks.collection_query` (см. раздел `collection_query`), чтобы она могла завершиться.
4. Успешно завершить Celery-задачу.

**Ошибки**

- Задача уже завершена: идемпотентная обработка (повторное сообщение из сети).
- Несогласованный UUID: логирование и дискардинг или ошибка — политика реализации.

## `tasks.evaluation`

**Назначение.** Запустить асинхронную оценку вакансии в **job-postings-evaluator** и дождаться события `evaluation-complete` (UC-02 п.5–6).

| Параметр             | Описание                                                                                 |
|----------------------|------------------------------------------------------------------------------------------|
| `jobPostingUuid`     | Внутренний UUID вакансии                                                                 |
| `evaluationTaskUuid` | UUID оркестрационной задачи `EVALUATION` (если уже создана; иначе создаётся в алгоритме) |
| `parentTaskUuid`     | Опционально; связь с `COLLECTION_QUERY` или другим родителем                             |

**Алгоритм работы:**

1. При необходимости создать оркестрационную задачу `EVALUATION` с указанием `parent_task_uuid`, статус `RUNNING`, в контексте — `jobPostingUuid`.
2. Вызвать **job-postings-evaluator**: `POST /evaluate/async/{jobPostingUuid}` с опциональным JSON-телом `AsyncSingleEvaluateOptions` (`evaluationTaskUuid`, `parentTaskUuid`), если корреляция с оркестратором нужна (см. OpenAPI evaluator).
3. Ожидать сигнал завершения, выставляемый обработчиком `tasks.evaluation_complete` для пары (`evaluationTaskUuid`, `jobPostingUuid`).
4. После сигнала: при необходимости обновить запись `EVALUATION` в хранилище и успешно завершить Celery-задачу.

**Ошибки**

- Ошибка вызова evaluator: статус `FAILED`, retry по политике.
- Таймаут ожидания `evaluation-complete`: аналогично `collection_query`.

## `tasks.evaluation_complete`

**Назначение.** Зафиксировать результат оценки и разблокировать ожидающую `tasks.evaluation` (UC-02 п.7).

| Параметр             | Описание                                    |
|----------------------|---------------------------------------------|
| `evaluationTaskUuid` | UUID оркестрационной задачи `EVALUATION`    |
| `executionLog`       | Лог выполнения                              |
| `result`             | Итог оценки (релевантность, статусы и т.д.) |

**Алгоритм работы:**

1. Проверить наличие оркестрационной задачи `EVALUATION` с `evaluationTaskUuid`.
2. Выполнить завершение в хранилище (`completion` с `SUCCEEDED` или `FAILED` в зависимости от содержимого `result` / доменных правил).
3. Снять блокировку ожидания с соответствующей `tasks.evaluation`.
4. Успешно завершить Celery-задачу.

**Ошибки**

- Идемпотентность при повторной доставке сообщения — как для `collection_query_complete`.

## `tasks.notification`

**Назначение.** Выполнить действие по уведомлению пользователя при релевантности выше порога (UC-02 п.7.1). Конкретный канал (e-mail, мессенджер, внутренняя лента) в этой спецификации не фиксируется.

| Параметр                | Описание                     |
|-------------------------|------------------------------|
| `jobPostingUuid`        | UUID вакансии                |
| `relevanceScore`        | Опционально                  |
| `notificationThreshold` | Опционально                  |
| `message`               | Опционально                  |
| `evaluationTaskUuid`    | Опционально; связь с оценкой |

**Алгоритм работы:**

Всегда завершает работу со статусом `FAILED` и сообщением "Not implemented"

---

## Порядок и конкурентность

- Все сообщения в очереди могут поступать **конкурентно**, воркеры должны быть stateless с идемпотентной записью в хранилище.
- Зависимости **parent-child** в оркестрационном хранилище не дублируют граф Celery: дочерние Celery-задачи могут завершаться раньше родительских; согласованность статусов пакета `COLLECTION_BATCH` обеспечивается отдельной агрегирующей логикой (см. [readme.md](./readme.md)).
