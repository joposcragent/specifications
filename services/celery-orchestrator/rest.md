# REST-интерфейс оркестратора

## Контроллер queue-broker

Операции только ставят сообщение в очередь Celery (или иным образом инициируют соответствующую Celery-задачу через клиент приложения). Успешный ответ **`HTTP 204`** означает: тело запроса принято, валидация пройдена, постановка в брокер выполнена (тело ответа пустое).

Согласованные **литералы имён Celery-задач** (для единообразия реализации и документации; при расхождении правятся в одном месте конфигурации приложения):

| Событие REST / домен        | Имя Celery-задачи (пример)        |
|-----------------------------|-----------------------------------|
| `collection-batch`          | `tasks.collection_batch`          |
| `collection-query`          | `tasks.collection_query`          |
| `collection-query-progress` | `tasks.collection_query_progress` |
| `collection-query-complete` | `tasks.collection_query_complete` |
| `evaluation`                | `tasks.evaluation`                |
| `evaluation-complete`       | `tasks.evaluation_complete`       |
| `notification`              | `tasks.notification`              |

### Запуск пакетного сбора

`POST /queue-broker/events/collection-batch`

| Входной параметр | Источник    | Описание                                                     |
|------------------|-------------|--------------------------------------------------------------|
| `triggeredAt`    | тело (опц.) | Момент инициации; если не передан — подставляется на сервере |
| `correlationId`  | тело (опц.) | Внешний идентификатор трассировки                            |
| прочие ключи     | тело (опц.) | Произвольные дополнительные поля верхнего уровня             |

Алгоритм работы:

1. При наличии тела проверяет типы и допустимость полей (`CollectionBatchEvent`).
2. Формирует kwargs для Celery-задачи `tasks.collection_batch`: при необходимости включает `triggeredAt`, `correlationId`; дополнительные ключи тела сверх фиксированных свойств схемы (`additionalProperties`) добавляет в тот же словарь kwargs без переименования полей и значений — они отображаются во **Flower** в карточке задачи (раздел kwargs).
3. Вызывает постановку задачи в брокер (например `apply_async` с маршрутизацией на очередь, заданную конфигурацией оркестратора).
4. При успехе возвращает **`HTTP 204`**.
5. При ошибке валидации тела возвращает **`HTTP 400`** с пояснением в `text/plain`.
6. При необработанном исключении возвращает **`HTTP 500`** с текстом исключения в теле ответа.

### Запуск сбора по одному поисковому запросу

`POST /queue-broker/events/collection-query`

| Входной параметр      | Источник | Описание                                                                               |
|-----------------------|----------|----------------------------------------------------------------------------------------|
| `searchQuery`         | тело     | Строка поискового запроса (аналог элемента списка из `GET /search-query/list` в UC-02) |
| `parentBatchTaskUuid` | тело     | UUID оркестрационной задачи пакета `COLLECTION_BATCH`                                  |

Алгоритм работы:

1. Проверяет обязательные поля и формат UUID.
2. Убеждается, что в хранилище оркестрационных задач существует задача с `uuid` = `parentBatchTaskUuid` и типом `COLLECTION_BATCH` (или иной согласованной политикой допускаемый родитель).
   1. Если не найдена — **`HTTP 404`** с текстом вида «родительская задача не найдена».
3. Формирует kwargs для `tasks.collection_query` (`searchQuery`, `parentBatchTaskUuid` и при необходимости идентификаторы для корреляции с записью дочерней задачи в БД).
4. Ставит задачу в брокер Celery.
5. При успехе — **`HTTP 204`**.
6. Ошибка валидации — **`HTTP 400`**; необработанное исключение — **`HTTP 500`**.

### Промежуточный прогресс сбора по запросу

`POST /queue-broker/events/collection-query-progress`

| Входной параметр          | Источник    | Описание                                       |
|---------------------------|-------------|------------------------------------------------|
| `collectionQueryTaskUuid` | тело        | UUID оркестрационной задачи `COLLECTION_QUERY` |
| `currentPage`             | тело (опц.) | Номер страницы выдачи                          |
| `totalPages`              | тело (опц.) | Известное число страниц                        |
| `executionLogFragment`    | тело (опц.) | Фрагмент лога                                  |
| `detail`                  | тело (опц.) | Произвольный JSON прогресса                    |

Алгоритм работы:

1. Валидирует тело; `collectionQueryTaskUuid` обязателен.
2. Проверяет наличие оркестрационной задачи с данным UUID (и при необходимости тип `COLLECTION_QUERY`).
   1. Если не найдена — **`HTTP 404`**.
3. Формирует kwargs для `tasks.collection_query_progress` и ставит задачу в брокер (UC-02: crawler между страницами; на стороне воркера может дополнительно обновляться прогресс в БД по диаграмме последовательности).
4. Успех — **`HTTP 204`**; валидация — **`HTTP 400`**; исключение — **`HTTP 500`**.

### Завершение сбора по запросу

`POST /queue-broker/events/collection-query-complete`

| Входной параметр          | Источник | Описание                           |
|---------------------------|----------|------------------------------------|
| `collectionQueryTaskUuid` | тело     | UUID задачи `COLLECTION_QUERY`     |
| `executionLog`            | тело     | Лог выполнения (строка или объект) |
| `result`                  | тело     | Структурированный итог сбора       |

Алгоритм работы:

1. Валидирует тело.
2. Проверяет существование задачи `collectionQueryTaskUuid`; при отсутствии — **`HTTP 404`**.
3. Ставит в брокер `tasks.collection_query_complete` с переданными логом и результатом (UC-02 п.3–4: сигнал завершения ожидания со стороны оркестратора).
4. Успех — **`HTTP 204`**; **`HTTP 400`** / **`HTTP 500`** — по общим правилам.

### Постановка оценки вакансии

`POST /queue-broker/events/evaluation`

| Входной параметр | Источник    | Описание                                     |
|------------------|-------------|----------------------------------------------|
| `jobPostingUuid` | тело        | Внутренний UUID вакансии в job-postings-crud |
| `parentTaskUuid` | тело (опц.) | Связь с родительской оркестрационной задачей |

Алгоритм работы:

1. Валидирует тело; `jobPostingUuid` обязателен.
2. Если передан `parentTaskUuid`, проверяет наличие родительской задачи; при отсутствии — **`HTTP 404`** (если включена проверка целостности).
3. Ставит в брокер `tasks.evaluation` (UC-02 п.5–6).
4. Успех — **`HTTP 204`**; **`HTTP 400`** / **`HTTP 500`** — по общим правилам.

### Завершение оценки вакансии

`POST /queue-broker/events/evaluation-complete`

| Входной параметр     | Источник | Описание                                 |
|----------------------|----------|------------------------------------------|
| `evaluationTaskUuid` | тело     | UUID оркестрационной задачи `EVALUATION` |
| `executionLog`       | тело     | Лог выполнения                           |
| `result`             | тело     | Итог оценки                              |

Алгоритм работы:

1. Валидирует тело.
2. Проверяет наличие задачи `EVALUATION` с `evaluationTaskUuid`; иначе — **`HTTP 404`**.
3. Ставит в брокер `tasks.evaluation_complete` (UC-02 п.7: снятие ожидания с оркестратора).
4. Успех — **`HTTP 204`**; **`HTTP 400`** / **`HTTP 500`** — по общим правилам.

### Уведомление о высокой релевантности

`POST /queue-broker/events/notification`

| Входной параметр        | Источник    | Описание                     |
|-------------------------|-------------|------------------------------|
| `jobPostingUuid`        | тело        | UUID вакансии                |
| `relevanceScore`        | тело (опц.) | Оценка релевантности         |
| `notificationThreshold` | тело (опц.) | Порог уведомления            |
| `message`               | тело (опц.) | Текст для канала уведомлений |
| `evaluationTaskUuid`    | тело (опц.) | Связь с задачей оценки       |

Алгоритм работы:

1. Валидирует тело; `jobPostingUuid` обязателен.
2. Ставит в брокер `tasks.notification` (UC-02 п.7.1).
3. Успех — **`HTTP 204`**; **`HTTP 400`** / **`HTTP 500`** — по общим правилам.

## Контроллер tasks

Операции ведут **записи** оркестрационных задач в хранилище сервиса. Они дополняют, но не заменяют постановку сообщений в Celery: вызывающая сторона (планировщик, воркер или другой микросервис) согласует порядок вызовов `tasks` и `queue-broker` в соответствии с UC-02.

### Создание оркестрационной задачи

`POST /tasks`

| Входной параметр | Источник    | Описание                                                                     |
|------------------|-------------|------------------------------------------------------------------------------|
| `taskUuid`       | тело (опц.) | Если задан — используется как UUID новой записи; иначе генерируется сервером |
| `type`           | тело        | `COLLECTION_BATCH`, `COLLECTION_QUERY`, `EVALUATION`                         |
| `parentTaskUuid` | тело (опц.) | UUID родителя в иерархии                                                     |
| `initialContext` | тело (опц.) | Начальный JSON-контекст                                                      |

Алгоритм работы:

1. Проверяет наличие обязательного поля `type` и допустимость значения.
2. Если задан `parentTaskUuid`, ищет родительскую задачу.
   1. Если не найдена — **`HTTP 404`**.
3. Если передан `taskUuid` и запись с таким UUID уже есть — **`HTTP 409`**.
4. Создаёт строку в хранилище: выставляет `status` = `PENDING` или `RUNNING` (по политике реализации), сохраняет `initial_context`, `parent_task_uuid`, временные метки.
5. Возвращает **`HTTP 201`** с телом `OrchestrationTaskCreated` (`taskUuid`, `type`, `status`).
6. Ошибка валидации — **`HTTP 400`**; необработанное исключение — **`HTTP 500`**.

### Получение задачи по UUID

`GET /tasks/{taskUuid}`

| Входной параметр | Источник      | Описание                    |
|------------------|---------------|-----------------------------|
| `{taskUuid}`     | path-параметр | UUID оркестрационной задачи |

Алгоритм работы:

1. Ищет задачу по `uuid` = `{taskUuid}`.
   1. Если не найдена — **`HTTP 404`**.
2. Возвращает **`HTTP 200`** с телом `OrchestrationTaskView` (тип, статус, родитель, контекст, последний прогресс, данные завершения при наличии).
3. Необработанное исключение — **`HTTP 500`**.

### Запись промежуточного результата

`POST /tasks/{taskUuid}/progress`

| Входной параметр       | Источник      | Описание               |
|------------------------|---------------|------------------------|
| `{taskUuid}`           | path-параметр | UUID задачи            |
| `recordedAt`           | тело (опц.)   | Метка времени события  |
| `executionLogFragment` | тело (опц.)   | Фрагмент лога          |
| `detail`               | тело (опц.)   | JSON деталей прогресса |

Алгоритм работы:

1. Ищет задачу по `{taskUuid}`; если нет — **`HTTP 404`**.
2. Если текущий статус задачи терминальный — **`HTTP 409`**.
3. Обновляет запись: сохраняет переданные поля прогресса (и при необходимости агрегирует лог), обновляет `updated_at`, при первом прогрессе может перевести статус из `PENDING` в `RUNNING`.
4. Успех — **`HTTP 204`** (пустое тело).
5. **`HTTP 400`** / **`HTTP 500`** — по общим правилам.

Связь с UC-02: события `collection-query-progress` в брокере и запись прогресса в БД на диаграмме последовательности согласуются с бизнес-логикой воркера; REST `tasks/.../progress` даёт прямой способ фиксации состояния без очереди, если так решено в реализации.

### Завершение задачи с фиксацией результата

`POST /tasks/{taskUuid}/completion`

| Входной параметр | Источник      | Описание                              |
|------------------|---------------|---------------------------------------|
| `{taskUuid}`     | path-параметр | UUID задачи                           |
| `terminalStatus` | тело          | `SUCCEEDED`, `FAILED` или `CANCELLED` |
| `executionLog`   | тело (опц.)   | Итоговый лог                          |
| `result`         | тело (опц.)   | Результат при успехе                  |
| `errorMessage`   | тело (опц.)   | Текст ошибки при `FAILED`             |

Алгоритм работы:

1. Ищет задачу; если нет — **`HTTP 404`**.
2. Если задача уже в терминальном статусе — **`HTTP 409`**.
3. Проверяет, что `terminalStatus` ∈ {`SUCCEEDED`, `FAILED`, `CANCELLED`}; иначе — **`HTTP 400`**.
4. Записывает `completion_payload`, выставляет статус = `terminalStatus`, обновляет `updated_at`.
5. Успех — **`HTTP 204`**.
6. **`HTTP 500`** при необработанном исключении.

После успешного завершения дочерних задач родительская задача `COLLECTION_BATCH` может оставаться в `RUNNING` до обработки всех дочерних `COLLECTION_QUERY` и их завершений — это правило агрегирования статуса задаётся реализацией и не дублируется в OpenAPI.
