# REST-интерфейс оркестратора Celery

Контроллер REST не содержит предметной логики: для эндпоинтов очереди он только ставит задачу в брокер **Celery** и перекладывает поля запроса в параметры создаваемой задачи; для эндпоинтов задач — только читает и возвращает уже сохранённое представление задачи.

## `POST /events-queue/progress`

| Входной параметр      | Источник     | Описание                |
|-----------------------|--------------|-------------------------|
| 📌 `{EventInputData}` | тело запроса | Объект `EventInputData` |

Алгоритм работы:

1. Безусловно и без всяких проверок ставит celery-задачу типа `task.progress` в очередь.
2. Все поля тела запроса передает в `kwargs` celery-задачи.
3. При успешной передаче сообщения брокеру возвращает **`HTTP 204`** без тела ответа.
4. При возникновении любого необработанного исключения возвращает **`HTTP 500`** с телом `text/plain` — текст исключения.

## `POST /events-queue/finish`

| Входной параметр            | Источник     | Описание                      |
|-----------------------------|--------------|-------------------------------|
| 📌 `{FinishEventInputData}` | тело запроса | Объект `FinishEventInputData` |

Алгоритм работы:

1. Безусловно и без всяких проверок ставит celery-задачу типа `task.finish` в очередь.
2. Все поля тела запроса передает в `kwargs` celery-задачи.
3. Добавляет в `kwargs` поля:
   1. `parentTaskResult` = `{FinishEventInputData}.result`;
   2. `parentTaskStatus` = `{FinishEventInputData}.status`.
4. При успешной передаче сообщения брокеру возвращает **`HTTP 204`** без тела ответа.
5. При возникновении любого необработанного исключения возвращает **`HTTP 500`** с телом `text/plain` — текст исключения.

## `POST /events-queue/{eventName}`

| Входной параметр      | Источник      | Описание                |
|-----------------------|---------------|-------------------------|
| 📌 `{eventName}`      | path-параметр | Имя события в path      |
| 📌 `{EventInputData}` | тело запроса  | Объект `EventInputData` |

Алгоритм работы:

1. Безусловно и без всяких проверок ставит celery-задачу, соответствующую значению `task.{eventName}`, в очередь.
2. Все поля тела запроса передает в `kwargs` celery-задачи.
3. При успешной передаче сообщения брокеру возвращает **`HTTP 204`** без тела ответа.
4. При возникновении любого необработанного исключения возвращает **`HTTP 500`** с телом `text/plain` — текст исключения.

## `GET /tasks/{taskUuid}`

| Входной параметр | Источник      | Описание                    |
|------------------|---------------|-----------------------------|
| 📌 `{taskUuid}`  | path-параметр | UUID оркестрационной задачи |

Алгоритм работы:

1. Запрашивает у слоя хранения представление задачи с UUID `{taskUuid}`.
2. Если данных по этому UUID нет, возвращает **`HTTP 404`** с телом `text/plain`.
3. Если данные есть, возвращает **`HTTP 200`** с телом `application/json` — объект по схеме `OrchestrationTaskView` из OpenAPI.
4. При возникновении любого необработанного исключения возвращает **`HTTP 500`** с телом `text/plain` — текст исключения.

## `GET /tasks/{taskUuid}/children`

| Входной параметр | Источник      | Описание                                 |
|------------------|---------------|------------------------------------------|
| 📌 `{taskUuid}`  | path-параметр | UUID родительской оркестрационной задачи |

Алгоритм работы:

1. Запрашивает у слоя хранения список дочерних задач для UUID `{taskUuid}`.
2. Если родительской задачи с таким UUID нет, возвращает **`HTTP 404`** с телом `text/plain`.
3. Если родительская задача есть, возвращает **`HTTP 200`** с телом `application/json` — массив объектов по схеме `OrchestrationTaskView` из OpenAPI.
4. При возникновении любого необработанного исключения возвращает **`HTTP 500`** с телом `text/plain` — текст исключения.
