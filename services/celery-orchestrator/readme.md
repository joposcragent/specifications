# Сервис celery-orchestrator

Python-сервис инкапсулирует взаимодействие с **Celery** и брокером **Redis**, а также ведёт учёт **оркестрационных задач** пайплайна сбора и оценки вакансий. Поведение бизнес-процесса задано в [UC-02 «Оркестрация процесса»](../../../solution/src/md/40-solution/10-use-cases/UC-02-collection-job.md); выбор стека описан в [ADR-003](../../../solution/src/md/40-solution/90-ADR/ADR-003-celery-redis.md).

Контракт REST API см. в [openapi.yaml](./openapi.yaml) в этом каталоге.

**Локальная разработка:** в `openapi.yaml` для localhost указан порт `8084`, чтобы не пересекаться с другими сервисами (например, crawler-headhunter на `8083`). Имя хоста в Docker-сети: `celery-orchestrator`, порт `8080`.

## Логическая модель хранения оркестрационных задач

Сервис ведёт записи задач в хранилище (конкретная СУБД и миграции определяются реализацией). Минимальный набор полей:

| Поле                 | Назначение                                                            |
|----------------------|-----------------------------------------------------------------------|
| `uuid`               | Первичный ключ задачи (совпадает с `taskUuid` в API)                  |
| `type`               | Тип: `COLLECTION_BATCH`, `COLLECTION_QUERY`, `EVALUATION`             |
| `parent_task_uuid`   | Ссылка на родительскую задачу или `NULL` для корня                    |
| `status`             | `PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `CANCELLED`              |
| `created_at`         | Время создания записи                                                 |
| `updated_at`         | Время последнего изменения                                            |
| `initial_context`    | JSON начального контекста (поисковый запрос, внешние UUID и т.д.)     |
| `progress_payload`   | Последний или агрегированный снимок прогресса (по решению реализации) |
| `completion_payload` | Итог: терминальный статус, лог, результат, текст ошибки               |

**Инварианты:**

1. Если задан `parent_task_uuid`, в хранилище должна существовать строка с `uuid` = этому значению.
2. Циклические ссылки в графе parent-child запрещены.
3. После перевода задачи в терминальный статус (`SUCCEEDED`, `FAILED`, `CANCELLED`) новые вызовы записи прогресса не принимаются (`HTTP 409`).
4. Повторное завершение уже завершённой задачи не допускается (`HTTP 409`).

Связь с UC-02: задача типа `COLLECTION_BATCH` порождает N задач `COLLECTION_QUERY` с `parent_task_uuid` на пакет; по мере работы crawler шлются события прогресса и завершения по запросу; после записи вакансий в CRUD ставятся задачи `EVALUATION`; по завершении оценки приходит событие завершения; при высокой релевантности — событие уведомления.
