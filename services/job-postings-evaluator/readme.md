# job-postings-evaluator

Сервис автоматического оценивания вакансий.

Работает с PostgreSQL:

- База данных `joposcragent`
- Схема `job_postings`
- Таблица `postings`

Сервис обращается по HTTP к `settings-manager` (порог релевантности и вектор эталонного контекста) и к `sentence-transformer` (векторизация текста и косинусное сходство векторов). Базовые URL задаются конфигурацией развёртывания; для `sentence-transformer` путь к API обычно имеет префикс, например `/sentence-transformer`, а вызовы идут на `POST` `{base}/text/vectorize` и `POST` `{base}/vectors/cosine-similarity`.

> **Состояние реализации.** Описанные ниже вызовы `celery-orchestrator` и заголовок `X-Joposcragent-correlationId` задают **целевой** контракт интеграции с пайплайном. В репозитории `job-postings-evaluator` соответствующий HTTP-клиент к оркестратору и чтение заголовка на момент ведения spec могут быть ещё не внедрены; поведение HTTP-эндпоинтов оценки без оркестраторной корреляции — как в шагах 1–7 разделов синхронных эндпоинтов, без пп. 8–9.

## `POST /evaluate/sync/list`

| Входной параметр  | Источник                                    | Описание                                                      |
| ----------------- | ------------------------------------------- | ------------------------------------------------------------- |
| 📌 `{list}`        | тело запроса, JSON-поле `list`              | Список внутренних UUID вакансий (`UuidsListRequest`)          |
| `{correlationId}`  | заголовок `X-Joposcragent-correlationId`   | Необязательный UUID родительского джоба в `celery-orchestrator` |

Алгоритм работы:

1. **Загрузка настроек (синхронно, строгая).** К `settings-manager` выполняются:
   1. `GET …/relevance-thresholds/CONTENT` — из тела ответа берётся числовой порог (`value`); обозначим `{threshold}`.
   2. `GET …/reference-context` — из тела ответа берётся непустой `vector`; обозначим `{reference_vector}`.
2. **Ошибка настроек.** Если порог не получен или `vector` отсутствует/некорректен — **HTTP 500**, `text/plain`, в теле **stack trace**; в лог пишется предупреждение.
3. **Отбор строк в `postings`.** Выбираются строки, где:
   1. `uuid` ∈ `{list}` (если `{list}` пуст, подходящих строк нет);
   2. `evaluation_status` ∈ {`NEW`, `PENDING`};
   3. сортировка по `uuid` (по возрастанию).
4. **Пустой отбор.** Если после п.3 не осталось ни одной строки — **HTTP 404** без тела.
5. **По каждой отобранной строке** (в порядке п.3):
   1. **Вектор контента `content_vector`.** Если в строке `content_vector` задан и не пуст, используется он (как вещественные числа). Иначе вызывается `POST` `{sentence-transformer-base}/text/vectorize` с полем `content`, равным полю `content` строки или пустой строке, если `content` = null. Возвращённый вектор используется далее. Ошибка HTTP/сети к sentence-transformer — исключение → **HTTP 500**, `text/plain` со **stack trace**.
   2. **Сходство с эталоном.** `POST` `{sentence-transformer-base}/vectors/cosine-similarity` с парой: `left` = вектор п.5.1, `right` = `{reference_vector}`. Из ответа берётся `similarity` (обозначим `{relevance}`).
   3. **Статус оценки.** Если `{relevance}` >= `{threshold}` — для строки итоговый статус `RELEVANT`, иначе `IRRELEVANT`.
   4. **Запись в БД** (отдельная фиксация в одной или нескольких операциях на строку, как в реализации). В `postings` у данной `uuid` обновляются: `evaluation_status`, `relevance` = `{relevance}` (с приведением к `float` хранимого типа), `content_vector` = вектор из п.5.1 (с приведением к массиву `float` для БД), `updated_at` = текущий момент.
6. **Успех.** **HTTP 200** и **JSON-массив** объектов `{ "uuid": "…", "status": "RELEVANT" | "IRRELEVANT" }` (по одному элементу на каждую **обработанную** в п.5 строку; порядок соответствует п.3).
7. **Неотловленное исключение** (после шагов 1–5 до формирования 200) — **HTTP 500** с `text/plain` и **stack trace**; в лог пишется ошибка.
8. **Если передан непустой `{correlationId}`** (целевой контракт, см. вступление; при **отсутствии** заголовка шаги 8–9 **не** выполняются):
   1. **При успехе (сформирован HTTP 200 по п.6),** в `celery-orchestrator` отправляется: `POST` `{orchestrator_base_url}/events-queue/finish` с телом JSON, минимум: `correlationId` = `{correlationId}`, `createdAt` = текущий момент времени, `status` = `SUCCEEDED` (строка), `result` = объект с кратким итогом, достаточным для снимка оркестрации, например `{ "evaluations": [ { "uuid": "…", "status": "RELEVANT" } ] }` в том же составе, что и HTTP-ответ по п.6. Поля `result`, `executionLog` и `status` в смысле оркестратора **не** передают в тело целевой Celery-задачи дословно как kwargs — снимок в Redis (см. реализацию `POST /events-queue/finish` в `celery-orchestrator`).
   2. **При п.2, п.4, п.7, или при сбое на п.5** (любой исход, ведущий к 4xx/5xx, до успешного 200), по аналогии с [созданием вакансии](../job-postings-crud/job-postings-controller.md) отправляется **уведомление о прогрессе** с сбоем: `POST` `{orchestrator_base_url}/events-queue/progress` с телом: `correlationId` = `{correlationId}`, `createdAt` = текущий момент, `status` = `FAILED`, `executionLog` = текст/описание ошибки; `jobPostingUuid` — идентификатор вакансии, **если** ошибка привязана к одной из строк отбора (например, сбой vectorize/cosine для неё), иначе **при необходимости** первый UUID из `{list}`. При наличии URL вакансии в БД — допустимо передавать `vacancyUrl` для согласования с другими вызовами `progress` в продукте. Ответ `celery-orchestrator` **204** при успехе.
9. **Некорректное тело запроса** (невалидный JSON, отсутствие обязательного поля `list`) — **HTTP 400** без детализации (как в типичном Spring-поведении для данных эндпоинтов); п.8–9 **не** выполняются.

## `POST /evaluate/sync/{jobPostingUuid}`

| Входной параметр     | Источник                                    | Описание                                                      |
| -------------------- | ------------------------------------------- | ------------------------------------------------------------- |
| 📌 `{jobPostingUuid}` | path-параметр                               | Внутренний UUID вакансии                                      |
| `{correlationId}`    | заголовок `X-Joposcragent-correlationId`   | Необязательный UUID родительского джоба в `celery-orchestrator` |

Алгоритм работы:

1. **Тот же конвейер, что в `POST /evaluate/sync/list`:** целевой набор в п.3 списка выше = **только** `{jobPostingUuid}`; шаги загрузки настроек, отбора, по-строчной оценки, записи в БД, коды 404/500 и формат ошибок — **как в пп. 1–5 и 7** раздела `sync/list`.
2. **Пустой отбор** (нет строки `NEW`/`PENDING` с данным `uuid`) — **HTTP 404** без тела.
3. **Успех.** **HTTP 200** и **JSON-объект** только с полем `"status"`: `RELEVANT` или `IRRELEVANT` (без поля `uuid` в теле ответа) — `SyncSingleEvaluationResponse`.
4. **Оркестратор (целевой контракт, см. вступление):** шаги ниже — **только** при непустом `{correlationId}`.
   1. **Успех (HTTP 200):** `POST` `{orchestrator_base_url}/events-queue/finish` с `correlationId`, `createdAt`, `status` = `SUCCEEDED`, `result` с минимум: `jobPostingUuid` = `{jobPostingUuid}` и `status` с итоговой оценки (при необходимости вложить `url` вакансии, если spec пайплайна требует).
   2. **Ошибка** в тех же смыслах, что **п.8(2)** раздела `sync/list`: `POST` `/events-queue/progress` с `status` = `FAILED`, `jobPostingUuid` = `{jobPostingUuid}`, `executionLog`, при наличии — `vacancyUrl` из БД; единообразно с `sync/list`.

## `POST /evaluate/async/list`

| Входной параметр | Источник         | Описание                                             |
| ---------------- | ---------------- | ---------------------------------------------------- |
| 📌 `{list}`       | тело, поле `list` | Список внутренних UUID вакансий, как в синхронном list |

**Поведение HTTP.** Сразу после приёма запроса (валидация тела — при нарушении **400**) возвращается **HTTP 200** с **пустым** телом. Дальнейшая обработка выполняется **асинхронно** в пуле потоков (например, с префиксом `evaluate-async-`).

**Фоновый алгоритм** (тот же **смысл** оценки, что и у `POST /evaluate/sync/list`, но с «мягкой» загрузкой настроек):

1. **Настройки.** `settings-manager` читается так же, как в синхронном пути; если порог эталонного вектора получить нельзя — **не** бросается в HTTP: пишется **WARN** в лог, **оценка не выполняется**.
2. **Отбор** строк `NEW`/`PENDING` по `list` — как в sync; **если пусто** — **WARN** в лог, завершение без HTTP-ошибки (клиент уже получил 200).
3. **По найденным строкам** — тот же цикл векторизация/косинус/сравнение/UPDATE, что и в sync, пока **не** влияя на уже отправленный ответ.

Исключения в фоне **не** превращаются в отдельный HTTP-ответ на исходный `POST` (клиент уже ушёл с 200). Они логируются согласно настройкам сервиса.

## `POST /evaluate/async/{jobPostingUuid}`

| Входной параметр     | Источник  | Описание                 |
| -------------------- | --------- | ------------------------ |
| 📌 `{jobPostingUuid}` | path      | Внутренний UUID вакансии |

**Поведение HTTP.** **HTTP 200** пустое тело сразу; далее **асинхронно** запускается тот же **алгоритм, что и у** `POST /evaluate/sync/{jobPostingUuid}` (одна вакансия в `list` фоновой оценки), с теми же **отличиями** по настройкам/логам/отсутствию HTTP-ошибок, что в разделе `async/list` выше.

## `POST /evaluate/async/batch`

| Входной параметр  | Источник  | Описание                                                 |
| ----------------- | --------- | -------------------------------------------------------- |
| 📌 `size`          | тело JSON | Целое число ≥ 1 (`BatchAsyncProcessingParametersRequest`) |

**Поведение HTTP.** **HTTP 200** пустое тело немедленно. Прямого **синхронного** аналога `GET/POST /evaluate/sync/batch` **нет**.

**Фоновый алгоритм (одна пакетная задача, не N отдельных вызовов `async/{uuid}`):**

1. Загрузка настроек **как у** async-эндпоинтов: при проблемах — WARN, **выход** без оценки.
2. Из `postings` выбираются **первые** `size` записей в статусе `NEW` или `PENDING`, **упорядоченные** по `uuid` по возрастанию, **без** фильтра «список UUID из тела».
3. Для всех отобранных записей **один** прогон общего кода оценки строк (тот же по смыслу, что **шаг 5** в `sync/list`): векторизация при пустом `content_vector`, косинус к эталону, порог, `UPDATE` строк.
4. **Итог по каждой** обработанной вакансии (uuid и итоговый статус) **пишется** в **лог** уровня **INFO** (а не в отдельные фоновые `POST` на каждую вакансию).

## Краткое сравнение sync и async

- **Sync** — клиент **ждёт** конца оценки; **404/500/400** возможны по правилам выше; **200** с **телом** JSON-результатов.
- **Async** — **всегда 200** сразу (кроме невалидного тела при валидации) и **только** пустой ответ: фактическое сделано/не сделано определяется **логами** и **состоянием БД**, а не HTTP ответа этих `POST`. Поведение настроек и пустого отбора **мягче** (WARN, пропуск), чем в sync.
