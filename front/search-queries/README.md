# Поисковые запросы

## Назначение и маршрут

- **Раздел меню:** Настройки → **Поисковые запросы**.
- **Маршрут (пример):** `/settings/search-queries`.

## Макет (wireframe)

```text
+------------------------------------------------------------------+
|  Заголовок «Поисковые запросы»    [ Создать ]                     |
+------------------------------------------------------------------+
|  Таблица / список:                                                |
|  Название | Запрос (обрезка) | Обновлён / создан | [Изм.] [Удал.] |
+------------------------------------------------------------------+
|  (Модалка создания/редактирования: name, query-string)            |
|  (Модалка подтверждения удаления)                                |
+------------------------------------------------------------------+
```

## Данные и API

Базовый URL: `VITE_SETTINGS_MANAGER_BASE_URL`. Контракт: [services/settings-manager/openapi.yaml](../../services/settings-manager/openapi.yaml); алгоритмы контроллера: [services/settings-manager/search-query.md](../../services/settings-manager/search-query.md).

### Семантика поля `query`

- В БД хранится **текст запроса в смысле query-string** — содержимое URL **после первого `?`** для выбранной страницы поиска на hh.ru (например `area=1&text=java`).
- Полная ссылка для открытия в браузере собирается как **`{VITE_HH_SEARCH_BASE_URL}?{query}`**, где **`VITE_HH_SEARCH_BASE_URL`** — базовый URL страницы поиска **без** завершающего `?` (по умолчанию при сборке можно задать `https://hh.ru/search/vacancy`).
- **Обратная совместимость:** в старых записях поле может содержать полный URL; UI при редактировании показывает только часть после `?`, при сохранении нормализует к query-string. Клик по ссылке в таблице: если значение уже начинается с `http(s)://`, открыть как есть; иначе собрать из базы и `query`.

### Список

- **GET /search-query/list**
- Ответ: JSON-массив элементов `SearchQueriesItem`: `uuid`, `name`, `query`, `createdAt`, `updatedAt`

### Создание

- Клиент генерирует **UUID v4** для сущности.
- **POST /search-query/{entityUuid}** с телом `application/json` `SearchQueryCreateRequest`: обязательные `name`, `query` (query-string или полный URL hh.ru для нормализации на клиенте).

### Обновление

- **PATCH /search-query/{entityUuid}** с телом `SearchQueryPatchRequest`: хотя бы одно из полей `name`, `query`.

### Удаление

- **DELETE** `/search-query/{entityUuid}`

### Получение одной записи (опционально для формы редактирования)

- **GET** `/search-query/{entityUuid}`

## Алгоритмы активных элементов

### Кнопка «Создать»

- **Клик:** открыть диалог с полями `name` (строка до 512 символов), `query` (query-string после `?` или полная ссылка hh.ru — перед отправкой нормализовать к query-string).
- **Подтверждение в диалоге:** сгенерировать `entityUuid`, вызвать `POST /search-query/{entityUuid}`; при **409** — сообщение, что UUID уже занят (редкий случай — сгенерировать новый uuid и повторить или показать ошибку).
- **Успех:** закрыть диалог, обновить список (`GET /search-query/list`).

### «Изменить» для строки

- **Клик:** открыть диалог с предзаполненными `name`, `query` (для отображения — нормализованная строка после `?`).
- **Сохранить:** `PATCH` с изменёнными полями; при **400** (ничего не передано) — показать ошибку валидации.
- **Успех:** закрыть диалог, обновить список.

### Колонка «Запрос» в таблице

- В ячейке показывают **первые 100 символов** значения **`query` из БД** (как хранится, без подстановки полного URL). При длине больше 100 — обрезка с многоточием.
- Текст ячейки — **кликабельная ссылка**: `href` = полная ссылка по правилам раздела «Семантика поля `query`», `target="_blank"`, `rel="noopener noreferrer"`.

### Колонка «Обновлён / создан»

- Одна дата: **`updatedAt`**, если непустое; иначе **`createdAt`**.

### «Удалить» для строки

- **Клик:** открыть **модальное подтверждение** («Удалить запись …?») с кнопками Отмена / Удалить.
- **Подтверждение удаления:** `DELETE /search-query/{entityUuid}`; при **404** — сообщение и обновление списка.
- **Успех:** закрыть модалку, обновить список.

## Открытие страницы (mount)

1. `GET /search-query/list`.
2. **200** и пустой массив — пустое состояние («нет запросов»), не ошибка.
3. **500 / сеть** — сообщение об ошибке, таблица без данных или с последним кэшем — по политике UX.

## Обновление страницы (F5)

Повторить `GET /search-query/list`; открытые модалки закрываются.
