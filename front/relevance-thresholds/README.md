# Пороги релевантности

## Назначение и маршрут

- **Раздел меню:** Настройки → **Пороги релевантности**.
- **Маршрут (пример):** `/settings/relevance-thresholds`.

## Макет (wireframe)

```text
+------------------------------------------------------------------+
|  Блок «Общий порог» (тип CONTENT)                                 |
|  Значение [____]   Обновлён / создан: дата (read-only)            |
|  [ Сохранить ]                                                    |
+------------------------------------------------------------------+
|  Блок «Порог для уведомлений» (тип NOTIFICATION)                  |
|  Значение [____]   Обновлён / создан: дата (read-only)            |
|  [ Сохранить ]                                                    |
+------------------------------------------------------------------+
```

Два блока **независимы**: у каждого своя кнопка сохранения и свой запрос к API.

## Данные и API

Базовый URL: `VITE_SETTINGS_MANAGER_BASE_URL`. Контракт: [services/settings-manager/openapi.yaml](../../services/settings-manager/openapi.yaml).

### Типы порогов (path-параметр `{type}`)

| UI-лейбл | `type` в API |
|----------|----------------|
| Общий порог | `CONTENT` |
| Порог для уведомлений | `NOTIFICATION` |

### Загрузка значений

Варианты (любой согласованный с реализацией):

- `GET /relevance-thresholds/list` → объект `RelevanceThresholdsList` с массивом `list` элементов `RelevanceThresholdItem`;
- или два вызова `GET /relevance-thresholds/{type}` для `CONTENT` и `NOTIFICATION`.

Элемент `RelevanceThresholdItem` содержит: `value` (number), `type`, `createdAt`, `updatedAt`.

### Сохранение одного порога

- **Метод:** `POST /relevance-thresholds/{type}`
- **Тело:** `application/json`, объект `{ "value": <number> }`
- **Успех 200:** тело в OpenAPI пустое по описанию — после успеха выполнить повторное чтение этого типа (`GET` по типу или обновить локально по согласованию; минимум — повторный `GET` для актуальных `updatedAt`).

## Алгоритмы активных элементов

### Поле ввода значения (каждый блок)

- Допустимый ввод: **вещественное число от 0,0 до 1,0 включительно** (тот же смысл, что **0.0…1.0** в JSON).
- **Перед отправкой:** если значение вне диапазона или не число — **не** вызывать API; показать валидационное сообщение у поля.
- Если масштаб хранения в БД когда-либо отличался от UI, согласовать отображение отдельно; постановка UI: диапазон 0–1 на форме.

### Кнопка «Сохранить» в блоке CONTENT

- **Клик:** при успешной валидации → `POST /relevance-thresholds/CONTENT` с `{ "value": ... }`.
- Блок NOTIFICATION при этом **не** трогать.

### Кнопка «Сохранить» в блоке NOTIFICATION

- Аналогично с типом `NOTIFICATION`.

### Read-only дата «Обновлён / создан»

- Показать **`updatedAt`**, если в `RelevanceThresholdItem` оно непустое; иначе **`createdAt`**. При отсутствии обеих — прочерк или согласованный пустой UX.

### Ошибки API

- **4xx/5xx / сеть:** сообщение пользователю; значение в поле можно оставить для повторной попытки.

## Открытие страницы (mount)

1. Показать индикатор загрузки в обоих блоках или общий.
2. Загрузить пороги (list или два GET).
3. Заполнить поля `value` и метки времени; при отсутствии записи для типа — пустое поле и сообщение «не задан» по UX или нули только если так зафиксировано бэкендом.

## Обновление страницы (F5)

Повторить загрузку; несохранённые правки в полях теряются.
