# Оболочка приложения (app shell)

## Назначение и маршрут

Общий каркас SPA: не отдельный маршрут, а layout, внутри которого рендерятся дочерние страницы (`RouterView`). Задаёт пропорции колонок, боковое меню, область контента и глобальные контролы (тема).

## Макет (wireframe)

```text
+------------------------------------------------------------------+
| [Nav 15%]              | [Content 85%]              [theme] [T] |
|                        |                                        |
|  Главная (крупно)      |  <RouterView />                        |
|    Дашборд             |                                        |
|    Оркестратор         |                                        |
|                        |                                        |
|  Настройки (крупно)    |                                        |
|    Эталонный контекст  |                                        |
|    Пороги...           |                                        |
|    Поисковые запросы   |                                        |
|                        |                                        |
+------------------------+----------------------------------------+
```

- Левая колонка: **15%** ширины viewport; правая: **85%**.
- В правом верхнем углу области контента (или на стыке с шапкой): переключатель **светлая / тёмная** тема (компонент Material 3).
- Пункты меню первого уровня — **увеличенный** кегль; вложенные — **меньший** кегль.

## Данные и API

Оболочка **не** вызывает бизнес-API сама по себе. HTTP-клиенты дочерних экранов используют базовые URL из переменных окружения (см. [корневой README](../README.md)).

- Axios (или эквивалент) настраивается с `baseURL` = значение соответствующей `VITE_*_BASE_URL` для каждого бэкенда.
- CORS обрабатывается на стороне `settings-manager` и `job-postings-crud`; фронт только отправляет запросы с ожидаемых origin.

## Алгоритмы активных элементов

### Иерархическое меню

- **Клик по родительскому разделу** (опционально): раскрыть/свернуть дочерние пункты, если реализовано аккордеоном; иначе родитель может быть только подписью без навигации.
- **Клик по дочернему пункту**: `router.push` на маршрут экрана; активный пункт подсвечивается по `route.name` / префиксу пути.
- **Пункт «Оркестратор»**: не внутренний роут — см. [orchestrator-flower/README.md](../orchestrator-flower/README.md).

### Переключатель темы

- **Клик**: переключить тему приложения (светлая ↔ тёмная) в согласовании с выбранной M3-библиотекой (например Vuetify `useTheme`).
- **При смене**: записать выбор в `localStorage` (стабильный ключ, например `joposcragent-theme`), чтобы при следующем визите тема восстановилась.

### Область контента

- Рендер дочернего маршрута без дополнительной логики.

## Открытие страницы (первый заход / полная загрузка приложения)

1. Прочитать `localStorage` по ключу темы; если значение валидно — применить тему до первого paint (по возможности), иначе тема по умолчанию (светлая или системная — зафиксировать в реализации единообразно).
2. Отобразить layout с меню и пустой/загружаемой областью контента.
3. Роутер активирует целевой дочерний маршрут (например `/` → дашборд); логика загрузки данных — в спецификации этого экрана.

## Обновление страницы (F5, повторный вход по URL)

1. Повторить чтение темы из `localStorage` (состояние Pinia сбрасывается).
2. Восстановить маршрут из URL; меню отразит активный пункт.
3. Дочерний экран сам выполняет свои запросы при `mount` (см. соответствующие `README.md`).
